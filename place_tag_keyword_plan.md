## 2. 핵심 문제 정의

네이버 플레이스에서 업체당 태그(키워드)를 최대 5개 설정할 수 있다.  
어떤 태그를 달아야 상위에 노출되는지 알기 어렵고, 경쟁사가 어떤 태그를 쓰는지 파악하는 방법이 없다.  
→ **경쟁사 상위 업체의 태그를 한 번에 분석해서, 가장 많이 쓰인 태그 순위를 알려준다.**

---

## 3. 사용자 플로우 (MVP)

```
경쟁사 플레이스 URL 최대 10개 입력
        ↓
서버에서 각 URL fetch + keywordList 파싱
        ↓
전체 태그 집계 → 빈도순 정렬
        ↓
순위 + 빈도수 결과 출력
        ↓
추천 태그 5개 복사 버튼
```

### UX 원칙
- 로그인 없이 바로 사용
- URL 입력 → 결과 출력까지 단순한 1페이지 구조
- 결과: **태그명 + 빈도수(N/10)** 만 표시 (심플하게)

---

## 4. 기능 범위

### MVP (v1) — 포함
- 네이버 플레이스 URL 다중 입력 (최대 10개)
- 서버사이드 태그 파싱 및 집계
- 빈도순 태그 랭킹 결과 출력
- 추천 태그 5개 한 번에 복사 버튼

### V2 이후 — 제외
- 리뷰 키워드 분석
- 카테고리 자동 추천
- 업체명으로 경쟁사 자동 탐색
- 히스토리 저장 / 회원 기능

## 5. 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | Next.js 14 (App Router) | 기존 스택 동일 |
| 파싱 방식 | 서버사이드 fetch + 정규식 | `keywordList` JSON 추출 |
| 파싱 대상 URL | `pcmap.place.naver.com/place/{업체ID}` | 소스 확인 완료 |
| 스타일링 | Tailwind CSS | |
| 배포 | Vercel 무료 플랜 | |
| DB | 불필요 (MVP 단계) | |

### 파싱 원리
네이버 플레이스 페이지 소스에 아래 형태로 태그 데이터가 포함되어 있음:

```json
"keywordList": ["태그1", "태그2", "태그3", "태그4", "태그5"]
```

서버에서 HTML을 fetch한 뒤 정규식으로 `keywordList` 값을 추출 → 집계.  
JS 렌더링 없이 파싱 가능 (Playwright 불필요, 확인 완료).

## 6. 리스크 및 대응 방안

| 리스크 | 수준 | 대응 |
|--------|------|------|
| 네이버 HTML 구조 변경 시 파싱 깨짐 | 중간 | 정기 모니터링, 알림 설정 |
| 서버 IP 반복 요청으로 차단 가능성 | 중간 | rate limiting, 요청 간 딜레이 처리 |
| 법적 리스크 (크롤링 이용약관) | 낮음 | 사용자가 URL 직접 입력하는 방식 → 자동 수집 아님 |
| keywordList가 일부 업체에 없는 경우 | 낮음 | 예외 처리 후 해당 URL 스킵 안내 |

## 7. 향후 수익화 방향 (검토 후보)

- **광고형**: 구글 애드센스 등 배너 광고
- **구독형**: 월정액으로 히스토리 저장, 업종별 태그 DB 제공

→ 트래픽 확보 후 사용 패턴 보고 결정

## 8. MVP 개발 우선순위

1. URL 입력 UI (텍스트에어리어, 최대 10개)
2. API Route: URL 받아서 keywordList 파싱 + 집계
3. 결과 화면: 태그 랭킹 + 빈도 바 차트
4. 복사 버튼 (추천 태그 5개)
5. 에러 처리 (잘못된 URL, 파싱 실패 등)
6. 반응형 모바일 대응
